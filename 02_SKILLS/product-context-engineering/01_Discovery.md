# Skill — Product Context Engineering

# 01 — Discovery

Versão: 1.0

---

# Objetivo

Discovery é a primeira etapa da metodologia Product Context Engineering.

Seu objetivo é compreender profundamente o problema antes de qualquer decisão sobre produto, tecnologia ou arquitetura.

Nesta fase não projetamos soluções.

Não escolhemos tecnologias.

Não escrevemos código.

Nós aprendemos.

---

# Princípio Fundamental

> Nunca projetar uma solução antes de compreender completamente o problema.

Toda implementação nasce de uma hipótese.

O Discovery existe para validar essa hipótese.

---

# Resultado Esperado

Ao final desta etapa devemos responder claramente:

- Qual problema estamos resolvendo?
- Quem possui esse problema?
- Como ele resolve isso atualmente?
- Por que as soluções atuais são insuficientes?
- Qual transformação queremos entregar?
- Vale a pena construir este produto?

Se alguma dessas respostas ainda não estiver clara, o Discovery não terminou.

---

# Entradas

O Discovery pode começar com qualquer uma das seguintes entradas:

- uma ideia;
- uma necessidade do mercado;
- uma dor observada;
- uma solicitação de cliente;
- uma oportunidade tecnológica;
- uma hipótese de negócio.

Toda entrada é considerada apenas uma hipótese.

Nunca uma verdade.

---

# Processo

## Etapa 1 — Definir o Problema

Descreva o problema em uma frase.

Exemplo:

"O processo atual exige muito trabalho manual."

Evite descrever soluções.

Errado:

"Precisamos criar um aplicativo."

Correto:

"As pessoas gastam muito tempo organizando informações."

---

## Etapa 2 — Identificar o Público

Responder:

Quem possui esse problema?

Quem sente essa dor diariamente?

Quem será impactado pela solução?

Quem não faz parte do público?

Quanto mais específico for o público, melhor.

---

## Etapa 3 — Entender o Cenário Atual

Documentar:

Como esse problema é resolvido hoje?

Quais ferramentas são utilizadas?

Quais limitações existem?

Quais custos esse problema gera?

---

## Etapa 4 — Levantar Hipóteses

Toda solução proposta deve ser tratada como hipótese.

Exemplo:

Hipótese:

"Automatizar este processo reduzirá o tempo gasto em 50%."

As hipóteses serão validadas posteriormente.

---

## Etapa 5 — Mapear Restrições

Identificar:

- restrições técnicas;
- restrições legais;
- restrições financeiras;
- restrições operacionais;
- restrições de prazo.

Projetos falham quando restrições são ignoradas.

---

## Etapa 6 — Definir Critérios de Sucesso

Responder:

Como saberemos que o projeto foi bem-sucedido?

Definir métricas objetivas.

Exemplos:

- redução de tempo;
- aumento de conversão;
- diminuição de erros;
- aumento de receita;
- satisfação do usuário.

---

# Perguntas Obrigatórias

Antes de concluir o Discovery, responda:

## Problema

Qual problema realmente existe?

Esse problema é frequente?

Quem sofre com ele?

---

## Usuário

Quem é o usuário?

Quem toma a decisão?

Quem paga?

Quem utiliza?

---

## Mercado

Existem soluções semelhantes?

Como elas resolvem esse problema?

Quais lacunas ainda permanecem?

---

## Valor

Por que alguém utilizaria nossa solução?

Qual benefício ela entrega?

Qual transformação ela gera?

---

## Negócio

Existe potencial de sustentabilidade?

O projeto pode evoluir?

Existe um modelo de receita viável?

---

# Artefatos Gerados

Ao final do Discovery devem existir, no mínimo:

- definição do problema;
- definição do público;
- mapa de hipóteses;
- critérios de sucesso;
- restrições identificadas;
- análise inicial do mercado.

Esses artefatos servirão como entrada para a etapa de Vision.

---

# O que evitar

Evite:

- escolher tecnologia;
- discutir arquitetura;
- escrever código;
- criar interfaces;
- definir banco de dados;
- assumir hipóteses como fatos.

Tudo isso pertence às próximas etapas.

---

# Critérios para Encerramento

O Discovery termina apenas quando:

- [ ] O problema está claramente definido.
- [ ] O público está identificado.
- [ ] As hipóteses estão documentadas.
- [ ] As restrições são conhecidas.
- [ ] Os critérios de sucesso foram definidos.
- [ ] Existe evidência suficiente para seguir.

Caso contrário, continuar pesquisando.

---

# Entregáveis

Ao concluir esta etapa devem ser criados:

- Problem Statement
- User Definition
- Hypothesis Map
- Success Metrics
- Discovery Report

Esses documentos serão utilizados pela etapa seguinte.

# Regra Final

Discovery não serve para encontrar respostas rápidas.

Serve para garantir que estamos resolvendo o problema certo antes de investir tempo, dinheiro e desenvolvimento na solução errada.
